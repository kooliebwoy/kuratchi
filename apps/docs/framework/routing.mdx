## Route conventions

Place `.kuratchi` files inside `src/routes/`. File paths become URL patterns.

```text
src/app.kuratchi                       -> document shell (<!DOCTYPE>, <html>, <head>)
src/routes/layout.kuratchi             -> shared fragment wrapping every page
src/routes/index.kuratchi              -> /
src/routes/about/index.kuratchi        -> /about
src/routes/items/index.kuratchi        -> /items
src/routes/blog/[slug]/index.kuratchi  -> /blog/:slug
```

Three file roles, one template syntax:

- **`app.kuratchi`** — the document shell. Lives at `src/app.kuratchi` (not inside `routes/`). Optional; the framework provides a default when absent.
- **`layout.kuratchi`** — a fragment that wraps every page. Nest `<segment>/layout.kuratchi` for section-specific chrome.
- **`index.kuratchi`** — the page for a segment. Dynamic segments use `[brackets]`.

All three use the same shape: top `<script>` + template + `<slot></slot>` (where applicable). See [Project structure](/framework/project-structure) for the full reference.

<Note>
**File extension:** Route files MUST end in `.kuratchi`. The compiler discovers routes by scanning for this extension — plain `.html` files are ignored (and should not be used for KuratchiJS pages). The `.kuratchi` extension signals "this is a KuratchiJS template with mixed server / client script regions" so editor tooling, the compiler, and agents can treat it correctly. Use `.html` only for true static HTML served from `src/assets/`.
</Note>

## Route execution model

Kuratchi pages are **server-rendered by default** and the top `<script>`
block is **isomorphic** — it runs once on the server during SSR and
again in the browser after the page loads. Author one script per route;
the framework handles the dual execution, SSR data hydration, and
dead-code stripping for you.

```html
<script>
  import { getItems } from '$server/items';
  const items = await getItems();

  function onDelete(id) {
    // Runs in the browser when a button is clicked.
  }
</script>

<ul>
  for (const item of items) {
    <li>
      {item.title}
      <button onclick={onDelete(item.id)}>Delete</button>
    </li>
  }
</ul>
```

What runs where:

- `await $server/*` functions execute on the server at render time. The
  resolved values are serialized into the HTML response and hydrated
  into the browser copy of the script — no second round-trip for SSR
  data.
- Helper functions (`function onDelete(id)`) and non-`await` top-level
  code run in both environments. The framework wraps DOM listeners
  automatically via the [`on<event>={fn(args)}` directive](/framework/client-interactivity#event-handler-directive).
- Top-level statements whose bodies contain `await` are stripped from
  the browser bundle — they already ran server-side.

Use `src/server/*` for private modules and backend logic. `$server/*`
functions called from the browser become RPC stubs that POST to a
framework-managed endpoint.

For the full hydration model, dispatch rules, and `__kuratchiReadData`
escape hatch, see [Client interactivity](/framework/client-interactivity).

## Second `<script>` for browser-only code

When you need browser-only code that shouldn't run on the server —
third-party widget boot, direct DOM inspection, a library that reads
`window` — add a second `<script>` block after the template. It's
bundled as a client-only module.

```html
<script>
  import { getItems } from '$server/items';
  const items = await getItems();
</script>

<ul>
  for (const item of items) {
    <li data-copy={item.secret}>{item.title}</li>
  }
</ul>

<script>
  import { mountCopyButtons } from '$lib/copy-buttons';
  mountCopyButtons();
</script>
```

Prefer the single-script model for everything that references SSR data.
The second `<script>` is an escape hatch for code that has no server
counterpart.

## Layouts

`src/routes/layout.kuratchi` is a **fragment** that wraps every page. Not a document — no `<!DOCTYPE>`, no `<html>`, no `<body>`. Those live in `app.kuratchi`.

```html
<script>
  import { url } from 'kuratchi:request';
  const current = url.pathname;
</script>

<header>
  <a href="/" class={current === '/' ? 'active' : ''}>Home</a>
  <a href="/items" class={current.startsWith('/items') ? 'active' : ''}>Items</a>
</header>

<main>
  <slot></slot>
</main>
```

### Nested layouts

Drop a `layout.kuratchi` at any directory depth. It wraps every
route under that directory, composed with the root layout outside
and the page inside.

```text
src/routes/
├─ layout.kuratchi             ← wraps every page
├─ index.kuratchi              ← /
└─ admin/
   ├─ layout.kuratchi          ← wraps /admin/*
   ├─ users/
   │  └─ index.kuratchi        ← /admin/users (root + admin layouts)
   └─ settings/
      ├─ layout.kuratchi       ← wraps /admin/settings/*
      └─ index.kuratchi        ← /admin/settings (root + admin + settings)
```

Composition at render time goes **innermost first**: the route's HTML
is wrapped by its nearest layout, the result is wrapped by the next
layout up, and so on until the root layout hands its output to the
app shell. That means an author writing `<slot></slot>` in a nested
layout receives the already-wrapped child markup — the inner layout
is closer to the content it's decorating than the outer one.

Each layout is a self-contained fragment: its top `<script>` runs
independently, it has its own SSR data, and it can import its own
`$server/*` modules. Two layouts in the same chain can import
`kuratchi:request` and read the same `searchParams` / `params` /
`pathname`; the framework plumbs request state into every layout and
page uniformly.

Use nested layouts for chrome that's scoped to a section — a
sidebar in `/admin`, a ticket-drawer overlay on `/board/*`, tab
navigation in `/settings` — without duplicating it across every
page underneath.

### App shell

The document shell (doctype, `<html>`, `<head>`, `<body>`) belongs in `src/app.kuratchi` — alongside `worker.ts` and `middleware.ts`, not inside `routes/`:

```html
<!DOCTYPE html>
<html lang="en">
<head>
  <meta charset="utf-8" />
  <title>My App</title>
</head>
<body>
  <slot></slot>
</body>
</html>
```

Kuratchi auto-injects a `<link rel="stylesheet">` for `src/app.css` here — don't author it yourself. See [Styling](/framework/styling).

Optional. Kuratchi synthesizes a minimal default shell when the file is absent.

## Template syntax

### Interpolation

```html
<p>{title}</p>
<p>{@html bodyHtml}</p>
<p>{@raw trustedHtml}</p>
```

### Conditionals

```html
if (items.length === 0) {
  <p>Nothing here yet.</p>
} else {
  <p>{items.length} items</p>
}
```

### Loops

```html
for (const item of items) {
  <li>{item.title}</li>
}
```

### Components

```html
<script>
  import Card from '$lib/card.html';
</script>

<Card title="Stack">
  <p>Live</p>
</Card>
```

Package components like `@kuratchi/ui/*` are optional. Plain HTML and local `src/lib/*.html` components are first-class.

### Component Props

Components receive props via a `props` object. Destructure in the component's `<script>` block:

```html
<!-- src/lib/card.html -->
<script>
  const { title, class: className = '', variant = 'default' } = props;
</script>

<div class="card {className}" data-variant={variant}>
  if (title) {
    <h2>{title}</h2>
  }
  <slot></slot>
</div>
```

**Props patterns:**
- Destructure from `props` with defaults: `const { title, size = 'md' } = props;`
- Access directly in template: `{props.title}` or `data-variant={props.variant}`
- Use `class:` for className (reserved word): `const { class: className = '' } = props;`
- Children go in `<slot></slot>` (renders as `props.children`)

## Client reactivity

Use `$:` inside a second, browser-only `<script>` block when you need
reactive updates driven by client-side state:

```html
<script>
  let users = ['Alice'];

  $: console.log(`Users: ${users.length}`);

  function addUser() {
    users.push('Bob');
  }
</script>
```

Rules:

- The top `<script>` is **isomorphic** — it runs on the server AND in
  the browser, with SSR-resolved data hydrated via
  [`__kuratchi_data`](/framework/client-interactivity#how-hydration-works).
- `$server/*` imports are the server/RPC escape hatch. Awaited on the
  server, RPC-stubbed in the browser.
- You do not need to author `type="module"` yourself — the framework
  adds it when the script has ES module imports.
- `$:` is available in browser-only scripts (the second `<script>`
  block) for reactive updates driven by mutation.
- Proxy-backed array and object mutation works when `$:` is active.
- You should not need `typeof document !== 'undefined'` guards in
  normal route code — the compiler strips SSR-only blocks from the
  browser bundle and helper functions run unconditionally on both
  sides.
