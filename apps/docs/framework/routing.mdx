## Route conventions

Place `.html` files inside `src/routes/`. File paths become URL patterns.

```text
src/routes/index.html              -> /
src/routes/about/index.html        -> /about
src/routes/items/index.html        -> /items
src/routes/blog/[slug]/index.html  -> /blog/:slug
src/routes/layout.html             -> shared layout
```

## Route execution model

Kuratchi pages are SSR by default.

The top-level route `<script>` is client-side by default.

When you import from `$server/*`, Kuratchi treats those functions as the server/RPC escape hatch.

For server work, use `$server/` imports, actions, and runtime hooks.

```html
<script>
  import { getItems } from '$server/items';
  const items = await getItems();
</script>

<ul>
  for (const item of items) {
    <li>{item.title}</li>
  }
</ul>
```

This is especially important for `$server/` imports. When you `await` a `$server/` function at the top level, Kuratchi runs that read during SSR and keeps your code in the same order you wrote it, even though the authored `<script>` stays client-first.

Use `src/server` for private modules and backend logic.

## Browser-only JavaScript

Use plain `<script>` when you want vanilla DOM behavior in the browser:

```html
<script>
  import { mountCopyButtons } from '$lib/copy-buttons';

  mountCopyButtons();
</script>

<button type="button" data-copy="hello">Copy</button>
```

Kuratchi adds `type="module"` for you when the script contains ES module imports.

Put reusable browser helpers in `src/lib` so they can be imported cleanly from your route markup.

## Layouts

`src/routes/layout.html` wraps every page. Use `<slot></slot>` where route content should render.

```html
<!DOCTYPE html>
<html lang="en">
<head>
  <meta charset="utf-8" />
  <title>My App</title>
</head>
<body>
  <nav>
    <a href="/">Home</a>
    <a href="/items">Items</a>
  </nav>
  <main>
    <slot></slot>
  </main>
</body>
</html>
```

## Template syntax

### Interpolation

```html
<p>{title}</p>
<p>{@html bodyHtml}</p>
<p>{@raw trustedHtml}</p>
```

### Conditionals

```html
if (items.length === 0) {
  <p>Nothing here yet.</p>
} else {
  <p>{items.length} items</p>
}
```

### Loops

```html
for (const item of items) {
  <li>{item.title}</li>
}
```

### Components

```html
<script>
  import Card from '$lib/card.html';
</script>

<Card title="Stack">
  <p>Live</p>
</Card>
```

Package components like `@kuratchi/ui/*` are optional. Plain HTML and local `src/lib/*.html` components are first-class.

### Component Props

Components receive props via a `props` object. Destructure in the component's `<script>` block:

```html
<!-- src/lib/card.html -->
<script>
  const { title, class: className = '', variant = 'default' } = props;
</script>

<div class="card {className}" data-variant={variant}>
  if (title) {
    <h2>{title}</h2>
  }
  <slot></slot>
</div>
```

**Props patterns:**
- Destructure from `props` with defaults: `const { title, size = 'md' } = props;`
- Access directly in template: `{props.title}` or `data-variant={props.variant}`
- Use `class:` for className (reserved word): `const { class: className = '' } = props;`
- Children go in `<slot></slot>` (renders as `props.children`)

## Client reactivity

Use `$:` inside client-side `<script>` blocks when you need client updates:

```html
<script>
  let users = ['Alice'];

  $: console.log(`Users: ${users.length}`);

  function addUser() {
    users.push('Bob');
  }
</script>
```

Rules:

- The top route `<script>` is client-side.
- `$server/*` imports are the server/RPC escape hatch.
- You do not need to author `type="module"` yourself for imported client scripts.
- `$:` is available when you want reactive browser updates inside client scripts.
- Proxy-backed array and object mutation works when `$:` is active.
- You should not need browser guards in normal route code.
