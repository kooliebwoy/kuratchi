## What `kuratchi build` generates

The compiler scans `src/routes/` and writes generated runtime files into `.kuratchi/`.

| File | Purpose |
| --- | --- |
| `.kuratchi/routes.ts` | Compiled routes, render functions, actions, and RPC handlers |
| `.kuratchi/worker.ts` | Stable Wrangler entrypoint re-exporting the router plus Worker classes |
| `.kuratchi/do/*.ts` | Generated Durable Object proxy modules |

That build artifact layout is the framework contract. Route files are source code. `.kuratchi` is generated output.

## Local loop

```bash
bun run dev
```

During development, keep these rules in mind:

- Pages are server-rendered by default.
- `src/server` is private server-only code.
- The top-level route `<script>` is browser/client code.
- Browser reactivity uses `$:` when you want reactive updates.
- `kuratchi:environment` is compiled for the current mode, so `dev` branches differ between `kuratchi dev` and `kuratchi build`.
- Wrangler should continue pointing to `.kuratchi/worker.ts`.

## Execution model

KuratchiJS is SSR by default:

- The page HTML renders on the server.
- The top-level route `<script>` is browser/client code.
- Form actions execute on the server.
- Server logic lives in `$server/*`, actions, and runtime hooks.

That keeps the default mental model closer to Worker request handling than SPA hydration.

## Mode-specific compilation

Kuratchi bakes the current mode into generated output.

- In `kuratchi dev`, `import { dev } from 'kuratchi:environment'` compiles to `true` in generated server/runtime code.
- In `kuratchi build`, the same import compiles to `false`.

That applies to route scripts, `src/server/*.ts`, and `src/server/runtime.hook.ts`.

Use this when you need explicit development-only behavior such as local auth bypasses, extra logging, or test fixtures.

## Production check

Before shipping, verify:

- `src/routes/layout.html` exists
- `src/routes/index.html` or another entry route exists
- `wrangler.jsonc` points at `.kuratchi/worker.ts`
- `bun run build` succeeds
