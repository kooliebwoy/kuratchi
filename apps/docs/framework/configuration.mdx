## Where config lives

A Vite-first KuratchiJS app has three config files and one request-lifecycle
entrypoint. Each has a distinct job:

| File | Owns | Edited by you |
| --- | --- | --- |
| `vite.config.ts` | The Kuratchi plugin install point | Only to add other Vite plugins |
| `wrangler.jsonc` | Cloudflare deployment (secrets, routes, custom bindings) | Yes, except auto-synced fields (DOs, queues, containers, assets) |
| `kuratchi.config.ts` | Framework integrations (ORM, auth, UI, CSS) | Yes, when you opt into any of them |
| `src/middleware.ts` | Per-request logic (`request` / `route` / `response` / `error`) | Yes |

See [Project structure](/framework/project-structure) for the full file reference.

`kuratchi.config.ts` is **not** where request interception lives. If you want
to gate routes, log requests, attach `ctx.locals`, route to agents, or shape
responses, use `src/middleware.ts`.

## `kuratchi.config.ts`

The config file is optional. You only need it when wiring framework integrations such as ORM, auth, CSS processing, or custom asset settings.

```ts
import { defineConfig } from '@kuratchi/js';
import { kuratchiOrmConfig } from '@kuratchi/orm/adapter';
import { kuratchiAuthConfig } from '@kuratchi/auth/adapter';

export default defineConfig({
  orm: kuratchiOrmConfig({
    databases: {
      DB: { schema: appSchema },
      NOTES_DO: { schema: notesSchema, type: 'do' },
    },
  }),
  auth: kuratchiAuthConfig({
    cookieName: 'kuratchi_session',
    sessionEnabled: true,
  }),
});
```

Without `kuratchi.config.ts`, the compiler falls back to framework defaults.

## `src/middleware.ts`

Middleware is separate from config. It runs per request and owns request-lifecycle
behavior.

```ts
import { defineMiddleware } from '@kuratchi/js';

export default defineMiddleware({
  auth: {
    async request(ctx, next) {
      return next();
    },
  },
});
```

Use middleware for:

- Auth gates and redirects
- Request logging
- Setting `ctx.locals`
- Agent routing
- Response/header shaping

See [Runtime](/framework/runtime) for the full middleware lifecycle.

## CSS and static assets

Kuratchi has two CSS paths — one convention-driven, one verbatim. Neither needs `kuratchi.config.ts`.

### `src/app.css` — the global stylesheet

Drop a file at `src/app.css`. Kuratchi auto-registers it as a client Rollup input, runs every Vite CSS plugin on it (Tailwind, PostCSS, CSS Modules, Lightning CSS…), and injects `<link rel="stylesheet">` into the app shell before `</head>`. You don't write the `<link>` tag and you don't configure anything.

```css
/* src/app.css */
@import "tailwindcss";
@plugin "daisyui";
@source "./routes/**/*.kuratchi";
```

### `src/assets/` — static files served verbatim

Files in `src/assets/` are served byte-for-byte by Wrangler's ASSETS binding at the URL root. No plugin transforms them. Use this for favicons, prebuilt CSS, images, fonts, JSON.

```text
src/assets/favicon.ico  → /favicon.ico
src/assets/hero.png     → /hero.png
```

Kuratchi keeps Wrangler's `assets.directory` in sync automatically. See [Styling](/framework/styling) for Tailwind setup, CSS frameworks, and migration notes.

## @kuratchi/ui Component Library

The `@kuratchi/ui` package provides server-rendered HTML components. It's independent of your CSS choice.

```ts
import { defineConfig } from '@kuratchi/js';
import { kuratchiUiConfig } from '@kuratchi/ui/adapter';

export default defineConfig({
  ui: kuratchiUiConfig({ 
    theme: 'dark',    // 'light' | 'dark' | 'system'
    radius: 'default' // 'none' | 'default' | 'full'
  }),
});
```

Use components in your routes:

```html
<script>
  import Card from '@kuratchi/ui/card.html';
  import Badge from '@kuratchi/ui/badge.html';
</script>

<Card>
  <Badge variant="success">Active</Badge>
  <p>Card content</p>
</Card>
```

## What belongs here

Use the config file for:

- ORM schema bindings
- Auth configuration
- UI component theming
- `assetsPrefix` (if you want `src/assets/` served at a prefix other than `/`)

## What does not belong here

Do not use `kuratchi.config.ts` for:

- Request middleware
- Route guards
- Response hooks
- Agent routing
- Per-request `locals` setup

Routing, Durable Object discovery, generated worker output, and CSS processing are handled outside this file — by the compiler and Vite plugins respectively. Request lifecycle behavior lives in `src/middleware.ts`.
