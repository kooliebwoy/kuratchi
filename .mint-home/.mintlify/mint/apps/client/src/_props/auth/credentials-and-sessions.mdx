## Credentials config

The credentials module is configured through the compiler from `kuratchi.config.ts`, then exposed through runtime helpers:

```ts
import {
  signUp,
  signIn,
  signOut,
  getCurrentUser,
  requestPasswordReset,
  resetPassword,
} from '@kuratchi/auth';
```

Supported credentials options include:

* `binding`
* `defaultRole`
* `sessionDuration`
* `minPasswordLength`
* `signUpRedirect`
* `signInRedirect`
* `signOutRedirect`
* `excludeFields`

## Signup

`signUp(formData)` expects at least:

* `email`
* `password`

It also supports:

* `name`
* `orgName` in organization-aware mode

The helper:

* validates the password length
* hashes the password with `AUTH_SECRET`
* creates the user
* logs activity
* sets redirect state in request locals

## Signin

`signIn(formData)` expects:

* `email`
* `password`

On success it:

* verifies the password hash
* creates a session token
* stores the hashed session token
* builds a signed session cookie
* records activity
* sets the configured redirect

## Signout

`signOut()` resolves the current session from the cookie, deletes it from storage, clears the cookie, and redirects.

## Current user lookup

`getCurrentUser()` reads the signed session cookie, validates the backing session record, and returns a safe user object with excluded fields removed.

By default, `passwordHash` is excluded.

## Password reset

The package includes:

* `requestPasswordReset(formData)`
* `resetPassword(formData)`

Reset requests:

* hash the reset token before storage
* store it in `passwordResets`
* send email through Resend when `RESEND_API_KEY` is available
* fall back to console output in development when email is not configured

Reset completion:

* validates the token
* updates the password hash
* invalidates active sessions
* deletes the reset record

## Cookie model

The session cookie is encrypted and signed through the auth crypto helpers. Session persistence uses the auth tables rather than trusting cookie data alone.
